# MailPilot — почтовый клиент для команды поддержки

Веб-приложение, которое превращает поток входящих обращений IT-поддержки в
управляемую очередь: письма автоматически классифицируются, размечаются
тегами по типу проблемы, ищутся «по смыслу» и сворачиваются в короткую
AI-сводку. Построено поверх базового классификатора писем и расширено
продвинутыми ML-фичами.

> Базовая классификация — это правила из исходного репозитория
> ([Email-super-67-classifier](https://github.com/MATI1n/Email-super-67-classifier-)).
> Продвинутый слой (AI-сводка, гибридный поиск, теги, веб-интерфейс) надстроен
> поверх него.

---

## Возможности

**Базовый функционал (как в исходном проекте)**
- Парсинг писем в разных форматах (заголовки `From/От кого`, `To/Кому`,
  `Subject/Тема`, `Date/Дата` на RU/EN).
- Классификация по категориям-папкам: `urgent`, `alerts`, `spam`,
  `hr_documents`, `newsletters`, `support` (fallback), `errors`.
- Устойчивость к «грязным» данным: бинарные файлы, картинки, повреждённый
  JSON, пустые письма не роняют систему, а уходят в категорию `errors`.

**Продвинутый слой (фичи на 9–10 баллов)**
1. **AI-суммаризатор** письма и заявки — на DeepSeek (`/chat/completions`).
   Возвращает краткую сводку, ключевые факты/сроки, рекомендованное действие
   и тональность. Без ключа работает локальный экстрактивный режим.
2. **Умный поиск** — гибридный ретрив: лексический канал **BM25** +
   семантический канал (векторная БД **Coderun**), результаты объединяются
   алгоритмом **Reciprocal Rank Fusion**. Без ключа Coderun семантический
   канал работает на локальных hashing-эмбеддингах.
3. **Теги по типу проблемы** (мульти-лейбл): Баги системы, Доступ и вход,
   Оплата и счета, Оборудование, Кадры, Срочный инцидент. Одно письмо может
   получить несколько тегов.

**Почтовый UX (по макету дизайнера)**
- Трёхколоночный интерфейс в стиле macOS Mail: папки/категории/теги · лента
  писем · карточка письма.
- Фильтры (Все / Непрочитанные / Избранное), пометки (звезда, архив,
  удаление с «Отменить»), счётчики, «умные» папки по категориям, поиск с
  подсветкой и бейджами каналов.

## Быстрый старт

```bash
cd mailpilot
./run.sh
```

Скрипт сам создаст виртуальное окружение, поставит зависимости, соберёт
фронтенд (если установлен Node.js) и поднимет сервер. Откройте
**http://localhost:8000**.

### Запуск вручную

```bash
# 1) Бэкенд
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2) Фронтенд (нужен Node.js 18+)
cd ../frontend
npm install && npm run build

# 3) Сервер (отдаёт API и собранный UI)
cd ../backend
uvicorn app.main:app --port 8000
```

### Dev-режим (горячая перезагрузка фронта)

```bash
# терминал 1 — бэкенд
cd backend && uvicorn app.main:app --reload --port 8000
# терминал 2 — фронтенд (Vite на :5173, проксирует /api на :8000)
cd frontend && npm run dev
```

## Подключение реальных DeepSeek и Coderun

Приложение работает «из коробки» в mock-режиме. Чтобы включить реальные
сервисы, скопируйте `backend/.env.example` в `backend/.env` и заполните ключи:

```env
DEEPSEEK_API_KEY=sk-...           # включает реальную AI-сводку
CODERUN_API_KEY=...               # включает семантический канал на Coderun
CODERUN_BASE_URL=https://...
```

Код менять не нужно — адаптеры (`ai.py`, `search.py`) сами переключаются на
сетевой режим при наличии ключей. Формы запросов к Coderun вынесены в класс
`_CoderunBackend` — при необходимости поправьте пути под фактический API.

## API (основные эндпойнты)

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/health` | статус, режим AI и вектора |
| GET | `/api/folders` | счётчики папок, категорий, тегов (сайдбар) |
| GET | `/api/emails?folder=&category=&tag=&unread=&starred=` | список писем |
| GET | `/api/emails/{id}` | письмо целиком (помечает прочитанным) |
| POST | `/api/emails/{id}/action` | `star/archive/delete/...` |
| POST | `/api/emails/{id}/summarize` | AI-сводка письма/заявки |
| GET | `/api/search?q=` | гибридный поиск (BM25 + семантика) |
| GET | `/api/stats` | сводная аналитика |

Интерактивная документация: **http://localhost:8000/docs**.

## Как устроена классификация и почему она устойчива

Категории определены из анализа датасета и практической пользы для
поддержки. Ключевая идея устойчивости — **многоуровневость и безопасный
fallback**:

1. Сначала формат/целостность файла (бинарь, картинка, битый JSON, пустое
   письмо → `errors`, с записью причины — система не падает на новых форматах).
2. Затем базовая категория по правилам; если ни одно не сработало — письмо не
   теряется, а попадает в `support` (fallback), то есть **новый, ранее
   невиданный тип обращения всегда оказывается в очереди поддержки**, а не
   выпадает из системы.
3. Поверх — независимый слой мульти-лейбл тегов по типу проблемы. Он не
   конкурирует с категорией и легко заменяется ML-моделью без изменения
   остального кода.

Такое разделение (формат → категория-папка → теги-проблемы) делает систему
расширяемой: добавление новой категории или тега — это правка одного списка,
а не переписывание пайплайна.
